# AGENTS

## Overview

This repository contains the **SnowballR mock backend**: a self-contained Node.js/TypeScript implementation of
the SnowballR gRPC API contract, used by the frontend for development and end-to-end tests without spinning up
the real backend and database. It does **not** persist data, does **not** implement real paper-fetching, and
its security posture is intentionally minimal — it must never be used in production. Prefer referencing the
README and the proto definitions over restating them here.

## Architecture and patterns

- **Stack:** Node.js + TypeScript, `@grpc/grpc-js` for the native gRPC server, `@grpc-web/proxy` for the
  gRPC-Web side. Logging via pino, deterministic randomness via seedrandom.
- **Two listeners:** native gRPC on `GRPC_PORT` (default 3000, for grpcui/grpcurl and native clients) and
  gRPC-Web on `GRPC_WEB_PORT` (default 3001, for the browser frontend).
- **Auth:** cookie-based tokens (`accessToken`, `refreshToken`) handled by `auth-interceptor.ts`. **Passwords
  are stored in plaintext** — this is by design for a mock; see the security warning in README.md.
- **Initial data:** controlled by the `EXAMPLE_DATA_FILE` env var, which references a TS file under `src/data/`
  that exports an object named `exampleData` conforming to the `ExampleData` interface (see README.md).
- **API contract sourcing:** `.proto` files come from the `api/` git submodule (the `snowballr-api` repo). To
  bump the API version, update the submodule. The generated stubs go under `src/grpc-gen/`.

## Boundaries

- Always do: prefer README references for usage; keep the in-memory data model honest to the proto contract;
  regenerate `src/grpc-gen/` only via `npm run compile:proto`; keep the security posture obviously-bad (no
  password hashing, etc.) so it cannot be mistaken for a real backend.
- Ask first: bumping the `api/` submodule pointer (changes the API contract this implements); changes to the
  auth interceptor or cookie format (frontend / e2e tests depend on it); changes to compose profiles or the
  Docker image; introducing real persistence; changes to `EXAMPLE_DATA_FILE` semantics or the `ExampleData`
  interface; CI workflow edits.
- Never do: ship or deploy this to production; commit secrets; add password hashing or TLS in a way that
  hides the mock's security posture; commit changes to `mock-backend.log`; edit files under `src/grpc-gen/` by
  hand; commit `dist/` or `node_modules/`.

## Commands (run from repo root)

### Build and run

- Update the API submodule: `git submodule update --init`
- Install deps: `npm install`
- Generate gRPC stubs: `npm run compile:proto`
- Build: `npm run build` (`tsc`)
- Run: `npm run start` (`node dist/main.js`)
- Run with dummy admin and sample data: `npm run dev`
- Watch mode: `npm run watch` (via nodemon)

### Lint / format

- Format: `npm run format` (Prettier)
- Lint: `npm run lint` (Prettier check + ESLint)

### Docker

- Default (empty data): `docker compose up`
- With dummy admin and sample data: `docker compose --profile mock-sample up`

### Dev shell (Nix)

- `nix develop` provides a reproducible dev shell (see `flake.nix`).

### Tooling (from README.md)

- `grpcui -plaintext 127.0.0.1:3000`
- `grpcurl -plaintext 127.0.0.1:3000 snowballr.SnowballR.IsAuthenticated`
- Auth header for grpcurl: `-H 'cookie:accessToken=<token>;refreshToken=<token>'`

## Environment variables (see README.md for the full table)

| Variable             | Default | Description                                                                           |
| -------------------- | ------- | ------------------------------------------------------------------------------------- |
| `GRPC_PORT`          | 3000    | Native gRPC server port.                                                              |
| `GRPC_WEB_PORT`      | 3001    | gRPC-Web proxy port.                                                                  |
| `GRPC_ALLOW_ORIGIN`  | `.*`    | Regex for CORS `Access-Control-Allow-Origin`.                                         |
| `ENABLE_DUMMY_ADMIN` | false   | Create a dummy admin user on startup (booleans: `1`/`yes`/`true`).                    |
| `RESPONSE_DELAY`     | 50      | Artificial response delay in milliseconds.                                            |
| `LOG_LEVEL`          | debug   | One of `fatal`, `error`, `warn`, `info`, `debug`, `trace`, `silent`.                  |
| `EXAMPLE_DATA_FILE`  | ""      | Relative path under `src/data/` to an `ExampleData` fixture (e.g. `standardData.ts`). |
| `RANDOMNESS_SEED`    | random  | Seed for pseudo-random generation; set for deterministic test runs.                   |

## Style, checks, and tests

- **Style:** Prettier + ESLint (`eslint.config.mjs`). Markdown follows `markdownlint.json`.
- **Tests:** No automated test suite in this repo — correctness is exercised by the consumers (frontend unit /
  integration / e2e tests run against this mock).

## Git and CI conventions

- CI runs build, code quality checks (`quality-check.yml`: markdown, lint, type check), and the deploy workflow
  on pushes to `main` (`deploy.yml` auto-increments the tag and publishes the Docker image to ghcr.io).

  (`submodules: "recursive"`).
